## Overview

Reverse API Engineer is a CLI tool that captures browser traffic (HAR files) and uses AI to automatically generate production-ready Python API clients. It supports three primary modes: manual browser capture, autonomous AI agent browsing, and re-engineering from previous captures.

## Dual-SDK pattern

The core of the project is a **dual-SDK architecture** controlled by the `sdk` field in `config.json`:

- **Claude SDK** (`engineer.py` → `ClaudeEngineer`): Uses `claude-agent-sdk` for direct Claude API integration
- **OpenCode SDK** (`opencode_engineer.py` → `OpenCodeEngineer`): Uses httpx to communicate with a local OpenCode server at `http://127.0.0.1:4096`

Both implementations inherit from `BaseEngineer` — an abstract base class that defines the common interface and shared prompt-building logic. SDK selection is determined at runtime from the `sdk` field in config.

| `engineer.py` | `ClaudeEngineer` — Claude SDK implementation |

| `messages.py` | Persists full conversation logs per run (JSONL format) |
| `pricing.py` | Model pricing database with three-tier fallback |

## Data flow

```
Browser (browser.py)
  └── Captures network traffic
        └── HAR saved to ~/.reverse-api/runs/har/{run_id}/

Engineer (engineer.py / opencode_engineer.py / copilot_engineer.py)
  └── Reads HAR, calls LLM
        └── Generates Python/JS/TS API client scripts
              ├── Permanent: ~/.reverse-api/runs/scripts/{run_id}/
              └── Local copy: ./scripts/{descriptive_name}/
```

1. The browser captures HAR traffic and saves it to `~/.reverse-api/runs/har/{run_id}/`
2. The engineer reads the HAR, sends it to the LLM, and streams back generated scripts
3. Scripts are written to both permanent storage and a local `./scripts/` directory with a human-readable name

## Data storage structure

```
~/.reverse-api/
├── config.json                    # User configuration
├── history.json                   # Run history with costs
└── runs/
    ├── har/{run_id}/              # Captured network traffic
    ├── scripts/{run_id}/          # Generated API clients
    ├── docs/{run_id}/             # Generated OpenAPI specs
    └── messages/{run_id}.jsonl    # LLM conversation logs
```

## Configuration system

Configuration lives at `~/.reverse-api/config.json`. The `ConfigManager` in `config.py` handles reading, writing, and backward-compatible migration of renamed keys.

```json
{
  "claude_code_model": "claude-sonnet-4-6",
  "opencode_provider": "anthropic",
  "opencode_model": "claude-sonnet-4-6",
  "sdk": "claude",
  "agent_provider": "browser-use",
  "browser_use_model": "bu-llm",
  "stagehand_model": "openai/computer-use-preview-2025-03-11",
  "output_dir": null
}
```

## Pricing system

Cost tracking uses a three-tier fallback:

1. **Local pricing** (`pricing.py`) — built-in prices for Claude and Gemini models (highest priority)
2. **LiteLLM pricing** — extended coverage for 100+ models (requires `[pricing]` optional extra)
3. **Default pricing** — falls back to Claude Sonnet 4.6 pricing when no match is found

The pricing system tracks input tokens, output tokens, cache creation/read tokens, and reasoning tokens for thinking models.

## Agent mode

Agent mode supports three provider options:

- **auto** (default): MCP-based browser automation with Claude Agent SDK. No additional installation required.
- **browser-use**: Multi-model browser automation. Requires the `[agent]` extra and browser-use installed from a specific git commit that includes HAR recording support.
- **stagehand**: Computer Use models from OpenAI and Anthropic.

API keys are read from environment variables: `BROWSER_USE_API_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`.

## Real-time sync

`sync.py` implements `FileSyncWatcher`, which watches generated script directories and copies files to the local `./scripts/` folder as they are written. Sync uses debounced file watching (500ms default) to avoid redundant copies. When enabled, you can see output files appear in real time as the LLM generates them.
